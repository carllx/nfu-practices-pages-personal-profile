# Task: Evaluate Interaction Burden

## 目的 (Purpose)

评估 IDE 内嵌学习交互设计的认知负担和对开发者工作流的潜在干扰，使用认知维度框架 (Cognitive Dimensions of Notations) 或类似 HCI 原则。

## 理论基础 (Theoretical Basis)

- **认知维度框架 (Cognitive Dimensions of Notations)** by Green & Petre: 提供了一套词汇来讨论信息 artifact (包括 IDE 交互) 的可用性，特别关注其对用户认知过程的影响。
    
- **开发者工具 HCI 原则 (HCI Principles for DevTools)**: 结合通用 HCI 原则与开发者特定需求 (例如: 最小化上下文切换、可预测性、键盘友好)。
    
- **参考资料**: `.bmad-core/data/hci-devtool-principles.md`
    

## 输入 (Inputs)

- `interaction_design`: 对要评估的交互设计的描述 (可以是文字描述、原型链接、甚至代码片段)。
    
- `context`: 交互发生的具体场景和开发者正在执行的任务。
    
- `target_ide`: (可选) 交互设计的目标 IDE 平台。
    

## 评估维度 (Key Dimensions to Evaluate)

[[LLM: 针对输入的设计，系统性地评估以下维度。为每个维度提供评分 (例如: 低/中/高 负担) 和具体理由。]]

1. **可见性 (Visibility)**:
    
    - 用户需要的信息或控件是否容易被看到？
        
    - 是否需要过多滚动、切换窗口或记忆信息？
        
    - **目标**: 高可见性，低查找负担。
        
2. **粘滞性 (Viscosity)**:
    
    - 做出小的、概念上简单的更改有多困难？
        
    - 是否需要执行许多繁琐的操作来完成一个简单意图？
        
    - **目标**: 低粘滞性，易于修改和实验。
        
3. **延展性/扩散性 (Diffuseness)**:
    
    - 表达一个概念需要多少“符号”或界面元素？
        
    - 界面是否过于冗长或稀疏？
        
    - **目标**: 适中的密度，既不拥挤也不空洞。
        
4. **隐含依赖/隐藏依赖 (Hidden Dependencies)**:
    
    - 一个部分的更改是否会对其他看似无关的部分产生意外影响？
        
    - 链接和关系是否清晰可见？
        
    - **目标**: 低隐藏依赖，关系明确。
        
5. **提前承诺 (Premature Commitment)**:
    
    - 用户是否需要在信息不全的情况下做出难以撤销的决定？
        
    - 是否强迫用户遵循固定的顺序？
        
    - **目标**: 低提前承诺，允许灵活性和探索。
        
6. **并列展示性 (Juxtaposability)**:
    
    - 是否可以轻松地并排比较相关信息？
        
    - **目标**: 高并列展示性，便于比较和关联。
        
7. **一致性 (Consistency)**:
    
    - 相似的操作或信息是否以相似的方式呈现？
        
    - 是否遵循 IDE 的通用模式和用户的现有习惯？
        
    - **目标**: 高一致性，降低学习成本。
        
8. **错误易发性 (Error-proneness)**:
    
    - 设计是否容易诱导用户犯错？
        
    - 是否提供了足够的预防和纠正错误的机制？
        
    - **目标**: 低错误易发性，高容错性。
        
9. **抽象梯度 (Abstraction Gradient)**:
    
    - 用户是否可以轻松地在不同抽象层级之间移动（例如，从高层概念到底层代码）？
        
    - 是否存在“抽象壁垒”难以跨越？
        
    - **目标**: 平滑的抽象梯度，支持不同层次的思考。
        
10. **角色表达性 (Role Expressiveness)**:
    
    - 界面元素的目的和作用是否清晰？
        
    - 用户是否容易理解每个部分的功能？
        
    - **目标**: 高角色表达性，易于理解界面结构。
        
11. **工作流中断 (Workflow Interruption)**:
    
    - **特别关注**: 交互是否强制打断了开发者当前的主要任务 (例如: 编码、调试)？
        
    - 上下文切换的成本有多高？
        
    - 交互是否可以被忽略或稍后处理？
        
    - **目标**: 最小化中断，尽可能融入现有工作流。
        

## 评估过程 (Evaluation Process)

1. **理解设计和上下文**: 仔细研究输入的设计描述和使用场景。
    
2. **逐一评估维度**: 针对每个维度，分析设计表现，并记录评分和理由。
    
3. **识别关键问题**: 找出负担最高或对工作流干扰最大的维度。
    
4. **提出改进建议**: 针对关键问题，提出具体的、可操作的改进方案，旨在降低认知负担和中断。
    
5. **生成评估报告**: 结构化地呈现评估结果和建议。
    

## 输出 (Output)

- 一份评估报告，包含：
    
    - 对每个维度的评分和详细分析。
        
    - 识别出的主要认知负担和工作流中断点。
        
    - 具体的改进建议列表，按优先级排序。
        
    - 总体评估结论 (例如: "负担过高，需重新设计", "基本可用，但需改进X和Y", "设计良好，负担较低")。
        

## 示例报告片段

```
### 评估报告: IDE 内联提示交互

**评估维度:**

* **可见性**: 中等负担。提示图标可见，但悬停才能看详情，可能错过。
* **粘滞性**: 低负担。修改代码通常会自动触发提示更新或消失。
* **一致性**: 高负担。提示样式与 IDE 原生 Linter 不一致，可能造成混淆。
* **工作流中断**: 中等负担。虽然可忽略，但频繁出现可能分散注意力。

**关键问题:**

1.  与原生 Linter 的一致性差，增加学习成本。
2.  潜在的信息过载和注意力分散。

**改进建议:**

1.  **[高优先级]** 统一提示样式，使其与 IDE 内建 Linter 风格保持一致。
2.  **[中优先级]** 增加配置选项，允许用户调整提示的触发频率或重要性阈值。
3.  **[低优先级]** 考虑将次要提示收敛到一个统一的“学习面板”而非内联显示。

**总体结论:** 基本可用，但一致性问题需优先解决以降低认知负担。
```
