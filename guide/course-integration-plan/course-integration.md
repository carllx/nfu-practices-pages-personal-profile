



# 课程整合方案：使用AI和开发工具构建个人网站

## 📋 课程基本信息

**课程名称**：从零到一：AI辅助的现代Web开发工作流（逆向工程版）  
**目标学员**：Web开发初学者、设计师转型开发者、个人品牌建设者  
**总时长**：3周课程（每周1次，每次约90分钟，课间作业3-7天）  
**核心成果**：一个结构完整、可持续维护的个人作品集网站

---

## 🎯 整合原则

基于对两份课程记录的深度分析和教学实践反思，本方案采用**逆向工程**教学法：

### 核心策略

1. **先快后慢**：第一周通过 AI Agents 快速生成网站初稿，让学生立即看到成果（激发兴趣和信心）
2. **逆向设计系统**：第二周从已生成的网站反推设计规范，在 Figma 中重建系统（比从零学设计更直观）
3. **Agent 协作流程**：学生通过与不同角色的 AI 对话，理解软件开发的完整流程
4. **认知负荷管理**：每节课聚焦一个核心能力，严格控制90分钟内完成核心交付物
5. **课间作业驱动**：强制性课间作业确保知识巩固和成果积累

## 📚 三周课程模块设计

### **第一周：AI Agents 驱动快速实现（看到成果）** 
**课堂时长**：90分钟  
**课间作业**：3-5天（核心成果期）  
**核心交付物**：可运行的网站初稿（HTML + CSS + JS）

#### 课堂环节（90分钟）

**破冰与基础设施搭建**
- **课程目标宣言**：今天就让你的网站"活"起来
- **最终成果展示**：一个完整的艺术家个人网站案例
  - 四个页面：首页、插画、产品、关于
  - 品牌色系统、交互效果
  - 强调：这是AI协作的成果，不是手工像素级编码

- **GitHub项目模板下载**
  - 应对访问不稳定的B计划：本地压缩包
  - 项目结构导览：agents/（最重要）、docs/、assets/
  - **实践**：每位学员下载并在VS Code中打开项目文件夹

- **Markdown快速入门（5分钟速成）**
  - 为什么用.md：AI最友好的格式，节省token成本
  - 核心语法：# 标题、** 粗体、- 列表、侧边预览
  - **实践**：在docs文件夹创建test.md体验一下

**AI智能体（Agent）的革命性概念**
- **痛点引入**：为什么普通AI输出不稳定？
  - 对比演示：同样的需求，通用AI vs Agent的输出差异
  
- **Agent = 角色化的专家 Prompt**
  - 不是聊天机器人，是你的虚拟团队成员
  - agents文件夹导览（打开几个.txt文件展示）
    - `analyst.txt`：分析师Mary，负责需求挖掘和项目简报
    - `pm.txt`：产品经理，负责PRD文档
    - `ux-expert.txt`：UX专家，负责前端规范
    - `architect.txt`：架构师，负责技术架构
    - `dev.txt`：前端开发者，负责写代码

- **关键约束：技术栈锁定**
  - ⚠️ **必须强调**：限制为纯 HTML + CSS + JavaScript
  - 原因：React/Vue对初学者是黑盒，无法理解和调试
  - **强制话术模板**：每次与AI对话必须包含
    ```
    技术限制：严格使用 HTML, CSS, JavaScript，
    不使用 React、Vue 等框架。
    ```

- **模型选择策略**（基于w9c1/w9c2的实践经验）
  - 讨论阶段：DeepSeek（深度思考模式）/ GPT / Gemini
  - 代码生成：智谱（在VS Code Agent模式中）
  - 原因：智谱适合执行，不适合头脑风暴

**与分析师Agent多轮对话（教师演示 + 学员实操）**

- **教师现场演示（15分钟）**
  - 打开DeepSeek/豆包，上传 analyst.txt
  - 第一轮对话：
    ```
    你好，Mary。我是一个插画艺术家，想创建个人作品网站。
    我喜欢1930年代橡皮管动画风格，有12件作品。
    技术限制：纯 HTML + CSS + JavaScript。
    请帮我进行初步分析。
    ```
  - 展示AI的反问（关于受众、色彩、功能等）
  - 第二轮：回答AI的问题，深入讨论
  - 第三轮：选择"头脑风暴"进一步构思
  - 最终轮：指令 "创建 brief.md 文档"

- **学员实操（15分钟）**
  - 每位学员用自己的创意与Analyst对话
  - 巡回指导：确保包含技术限制声明
  - 鼓励至少3轮对话，不要急于生成文档

**课堂总结与作业布置**
- **今天建立的核心能力**
  - GitHub项目管理
  - Markdown文档编写
  - 与AI Agent的结构化对话

- **关键要点回顾**
  - Agent不是万能的，需要你提供清晰的约束
  - 技术栈限制必须在第一轮对话就声明
  - 多轮对话比一次性生成质量更高

#### 课间作业（3-5天完成）⭐

这是整个课程最关键的产出期！学员需要通过与不同AI Agent对话，生成完整的网站初稿。

**必须完成（核心任务）**：

---

#### 任务1：与 Analyst Agent 对话 → 生成 `docs/brief.md`

**核心要点**：

用户只需提供基本需求，例如：
- "我想为艺术家 carllllllllx 创建一个个人作品集网站"
- 明确风格要求（如"1930年代橡皮管动画风格"）
- 指定技术限制（如"纯 HTML/CSS/JavaScript"）
- "如果有不确定的地方请你主动询问我" 利用这种机制及时澄清模糊需求

**明确项目需求**：向analyst agent描述你的项目基本信息，如"为艺术家创建橡皮管风格作品集网站"

![Analyst Agent 对话示例](https://i.imgur.com/otaaPN5.webp)

**操作步骤**：

1. **准备工作**
   - 打开 DeepSeek / ChatGPT / Gemini（推荐使用深度思考模式）
   - 在对话框中上传或粘贴 `agents/analyst.txt` 文件内容

2. **初始对话模板**
   ```
   你好，Mary（分析师的名字）。
   
   我想为[你的身份，如：插画艺术家/摄影师/设计师]创建一个个人作品集网站。
   
   风格要求：[描述你喜欢的风格，如：极简主义/复古/现代科技感]
   
   技术限制：严格使用纯 HTML、CSS、JavaScript，
   不使用 React、Vue 等框架。
   
   如果有不确定的地方请你主动询问我。
   
   请帮我进行初步分析。
   ```

3. **多轮对话要点**
   - Analyst会主动提问（目标受众、作品数量、色彩偏好等）
   - 认真回答每个问题，提供具体信息
   - 可选择"头脑风暴"进一步探索创意方向
   - ⚠️ 每次提到技术需求时，都要重申"纯 HTML/CSS/JavaScript"

4. **最终生成指令**
   ```
   现在让我们创建项目简报文档 brief.md
   ```

5. **验收标准**
   - brief.md 包含：项目概述、目标用户、风格定位、技术约束、设计方向

---

#### 任务2：与 PM Agent 对话 → 生成 `docs/prd.md`

**核心要点**：

上传项目简报文件：
- brief.md（项目简报）

使用话术：
> "我们现在需要生成 prd.md，我提供了一个和分析师讨论的结果就是这里的 brief.md"

⚠️ **重要提示**：当 PM 提出生成 Epic 列表时可以终止，强行生成 prd.md

![PM Agent 对话示例](https://i.imgur.com/tH50HZU.webp)

**操作步骤**：

1. **准备工作**
   - 上传或引用 `agents/pm.txt`
   - 准备好上一步生成的 `brief.md`

2. **初始对话模板**
   ```
   你好，我们现在需要生成产品需求文档 prd.md。
   
   我已经和分析师讨论过项目，生成了 brief.md（见附件）。
   
   请基于这份简报，帮我制定详细的产品需求文档。
   
   技术限制：纯 HTML、CSS、JavaScript。
   ```

3. **交互要点**
   - 当PM提出生成"Epic列表"（史诗列表）时，可以选择：
     - 继续深入讨论Epic
     - 或直接要求："现在直接生成完整的 prd.md"
   - PM可能会询问具体功能优先级，根据你的需求回答

4. **验收标准**
   - prd.md 包含：功能列表、用户故事、页面结构、交互逻辑

---

#### 任务3：与 UX Expert Agent 对话 → 生成 `docs/front-end-spec.md`（初稿）

**核心要点**：

上传项目简报文件：
- brief.md（项目简报）
- prd.md（产品需求文档）

使用话术：
> "我们制定设计规范的文档 front-end-spec.md"

明确表达具体需求：
- "我们的设计仅支持电脑端浏览器"
- Agent 会根据反馈调整相应部分的设计策略

![UX Expert Agent 对话示例](https://i.imgur.com/EdDr9aH.webp)

**操作步骤**：

1. **准备工作**
   - 上传或引用 `agents/ux-expert.txt`
   - 准备 `brief.md` 和 `prd.md`

2. **初始对话模板**
   ```
   你好，我们需要制定设计规范文档 front-end-spec.md。
   
   我已经完成了项目简报（brief.md）和产品需求文档（prd.md）。
   
   请基于这些文档，帮我制定前端设计规范。
   
   特别说明：
   - 我们的设计仅支持电脑端浏览器（桌面端）
   - 技术限制：纯 HTML、CSS、JavaScript
   ```

3. **交互要点**
   - 明确告知是否需要移动端适配（初学者建议先不做）
   - 如果Agent建议复杂的设计工具，可以要求简化
   - 重点关注：颜色系统、字体、间距、组件规范

4. **验收标准**
   - front-end-spec.md 包含：设计令牌（颜色、字体、间距）、组件规范、布局系统

⚠️ **重要提示**：这是初稿版本，第二周会在Figma中规范化和完善

---

#### 任务4：与 Architect Agent 对话 → 生成 `docs/architecture.md`

**核心要点**：

上传基础文档：
- brief.md（项目简报）
- prd.md（产品需求文档）
- front-end-spec.md（前端规范）

使用话术：
> "让我们来制定完整的 docs/architecture.md"

**具体限制要求**：
- "不需要 Netlify 部署流程，开发团队是初学者"
- 移除复杂构建工具（如 Vite、React 等），改用纯 HTML/CSS/JS
- 简化为静态托管，适合初学者团队

![Architect Agent 对话示例](https://i.imgur.com/c1tixYZ.webp)

**操作步骤**：

1. **准备工作**
   - 上传或引用 `agents/architect.txt`
   - 准备所有前面生成的文档（brief.md, prd.md, front-end-spec.md）

2. **初始对话模板**
   ```
   你好，让我们来制定完整的技术架构文档 architecture.md。
   
   我已经完成了：
   - 项目简报（brief.md）
   - 产品需求文档（prd.md）
   - 前端设计规范（front-end-spec.md）
   
   请基于这些文档，帮我制定技术架构。
   
   特别限制：
   - 开发团队是初学者
   - 不需要 Netlify 等复杂部署流程
   - 不使用构建工具（Vite、Webpack等）
   - 不使用前端框架（React、Vue等）
   - 使用纯 HTML、CSS、JavaScript
   - 最终部署到 GitHub Pages
   ```

3. **交互要点**
   - 如果Architect建议复杂工具链，及时制止
   - 强调："我们需要最简单、最基础的技术方案"
   - 关注：文件结构、命名规范、代码组织

4. **验收标准**
   - architecture.md 包含：项目文件结构、技术选型说明、开发流程、部署方案

---

#### 任务5：准备素材 → 创建 `assets/README.md`

**操作步骤**：

1. **收集作品素材**
   - 准备 5-10 张你的作品图片
   - 图片格式：JPG、PNG、WebP（推荐WebP，文件更小）
   - 图片命名：work-001.jpg, work-002.jpg ...
   - 放入 `assets/images/` 文件夹

2. **创建素材描述文档**
   - 在 `assets/` 文件夹创建 `README.md`
   - 为每张图片写简短描述：
     ```markdown
     # 作品素材说明
     
     ## work-001.jpg
     - 标题：[作品名称]
     - 创作年份：2024
     - 描述：[简短描述]
     - 标签：插画, 数字艺术
     
     ## work-002.jpg
     ...
     ```

3. **（可选）创建数据文件**
   - 可以让AI帮你生成 `assets/data.js`
   - 格式示例：
     ```javascript
     const artworks = [
       {
         id: "work-001",
         title: "作品名称",
         image: "./assets/images/work-001.jpg",
         description: "作品描述",
         year: 2024,
         tags: ["插画", "数字艺术"]
       }
     ];
     ```

#### 任务6：与 Dev Agent 对话 → 生成网站初稿代码

**操作步骤**：

1. **准备工作**
   - 打开 VS Code
   - 打开你的项目文件夹
   - 确保所有文档都已生成（brief, prd, front-end-spec, architecture）

2. **使用智谱 AI Agent 模式**
   - 在 VS Code 中打开智谱 AI 插件
   - 切换到 "Agent" 模式（不是 "Ask a question"）
   - 打开 `agents/dev.txt` 文件

3. **引用文档并下达指令**
   - 点击输入框旁边的 `@` 符号
   - 依次引用文件：
     - @brief.md
     - @prd.md
     - @front-end-spec.md
     - @architecture.md
     - @assets/README.md（如果有）
   
   - 输入指令：
     ```
     基于以上所有文档，请帮我生成个人作品集网站的代码。
     
     请依次生成：
     1. index.html（首页）
     2. style.css（样式文件）
     3. script.js（交互脚本）
     
     技术要求：纯 HTML、CSS、JavaScript，使用语义化标签。
     ```

4. **分步生成与测试**
   - Dev Agent会逐步生成代码
   - 每生成一个文件，就在浏览器中测试
   - 如果遇到问题，描述问题让Agent修正
   - 可以继续要求生成其他页面（about.html, gallery.html等）

5. **验收标准**
   - 能在浏览器中打开 index.html
   - 页面显示正常，无明显错误
   - 基本交互功能可用（导航、图片展示等）

### 整体作业要点总结

**核心原则**：
1. **顺序执行**：按照 Analyst → PM → UX Expert → Architect → Dev 的顺序
2. **文档传递**：每一步的输出是下一步的输入
3. **反复强调**：在每次对话中都要提及"纯 HTML/CSS/JavaScript"限制
4. **利用澄清机制**：在对话中加入"如果有不确定的地方请主动询问我"
5. **不追求完美**：这是初稿，目标是快速看到成果

**常见问题处理**：
- 如果AI建议使用框架：立即制止，重申技术限制
- 如果生成内容过于复杂：要求简化为初学者可理解的版本
- 如果某个Agent卡住：可以跳过，先完成其他步骤

**提交要求**：
- 所有文档齐全（brief.md, prd.md, front-end-spec.md, architecture.md）
- 至少有一个可运行的HTML页面
- 将整个项目文件夹压缩，准备第二周展示

### **第二周：逆向设计系统（从网站反推规范）** 
**课堂时长**：90分钟  
**课间作业**：5-7天  
**核心交付物**：规范化的 Figma 设计系统 + 更新的设计文档

#### 课堂环节（90分钟）

**成果展示与问题诊断**
- **学员网站初稿展示（3-4位）**
  - 在浏览器中打开展示
  - 庆祝：你们已经有了一个可运行的网站！
  - 集体观察：每个网站的视觉风格、颜色使用

- **引入核心问题**
  - 假设你现在想改变主色调，需要在代码中改多少个地方？
  - 如果要增加一个新页面，如何保持风格统一？
  - **痛点可视化**：现场演示维护噩梦（颜色散落在各处）

**逆向工程：从网站提取设计系统**

- **什么是逆向工程**
  - 不是从零学设计，而是从已有成果反推规则
  - 目标：为网站建立"设计宪法"，实现可维护性

- **Design Tokens（设计令牌）核心概念**
  - Token = 设计决策的变量化存储
  - 类比：代码中的常量，音乐中的音符
  - 现场演示（打开教师的Figma案例）：
    - 修改一个颜色Token，整个设计瞬间更新
    - 这就是系统化设计的力量

- **从你的网站中识别 Tokens**
  - **练习**：打开你的网站初稿，用笔记下：
    1. 用了几种颜色？（主色、点缀色、背景色、文字色）
    2. 用了几种字体大小？
    3. 间距规律是什么？（padding、margin）
  - 学员分享发现（2-3位）

**在 Figma 中重建设计系统（教师演示 + 学员实操）**

- **Figma 变量系统速成（10分钟）**
  - 打开空白Figma文件
  - 点击右侧面板的"Variables"（变量）
  - 创建颜色变量库：
    - `color-primary`
    - `color-accent-dark`（点缀深色）
    - `color-accent-light`（点缀浅色）
    - `color-bg-light`, `color-bg-dark`
    - `color-text-primary`, `color-text-secondary`
  - 创建间距变量：`spacing-xs/sm/md/lg/xl`
  - 应用变量到元素：选择元素 → Fill → 选择变量而非颜色值

- **学员实操：重建自己的设计系统（15分钟）**
  - 在Figma中创建变量库
  - 将第一周识别的颜色、间距录入为变量
  - 巡回指导：帮助解决Figma操作问题

**组件化思维与原子设计**

- **原子设计理论简介（5分钟）**
  - 原子（Atoms）：Token本身
  - 分子（Molecules）：按钮、输入框
  - 有机体（Organisms）：导航栏、卡片组
  - 模板（Templates）：页面布局
  - 页面（Pages）：实际内容

- **创建核心组件（15分钟）**
  - **教师演示**：创建标准按钮组件
    - 使用变量作为填充色和文字色
    - 定义多种状态（default, hover, active, disabled）
    - 使用Auto Layout控制内部间距
  
  - **学员实操**：
    - 尝试创建一个按钮或卡片组件
    - 使用刚才创建的Token变量

**课堂总结与作业布置**

- **今天学到的核心能力**
  - 逆向分析网站的设计规律
  - 在Figma中用变量系统管理设计
  - 组件化思维的初步建立

- **关键洞察**
  - 先实现后规范，比从零学设计更有效
  - 设计系统不是为了好看，是为了可维护
  - Token是设计与代码之间的桥梁

### **第三周：基于设计系统的迭代开发与部署**
**课堂时长**：90分钟  
**课间作业**：持续迭代  
**核心交付物**：部署上线的完整网站 + 系统进化计划

#### 课堂环节（90分钟）

**设计系统成果展示**
- **学员 Figma 展示（2-3位）**
  - 展示变量系统（颜色、间距）
  - 展示核心组件（按钮、卡片）
  - 同伴互评：这个设计系统是否清晰可维护？

- **教师点评要点**
  - Token命名是否语义化
  - 组件是否有清晰的状态定义
  - 系统是否便于扩展

**应用设计系统优化代码**

- **CSS 变量系统（Design Tokens 的代码化）**
  - 教师演示：将 Figma Token 转换为 CSS 自定义属性
  - 在 `style.css` 顶部定义：
    ```css
    :root {
      /* 颜色系统 */
      --color-primary: #1976D2;
      --color-accent-dark: #D32F2F;
      --color-bg-light: #FFFFFF;
      --color-text-primary: #212121;
      
      /* 间距系统 */
      --spacing-xs: 4px;
      --spacing-sm: 8px;
      --spacing-md: 16px;
      --spacing-lg: 24px;
      --spacing-xl: 32px;
      
      /* 字体系统 */
      --font-size-h1: 48px;
      --font-size-body: 16px;
    }
    ```

- **重构现有代码**
  - 找到所有硬编码的颜色值（如 `#D32F2F`）
  - 替换为变量（`var(--color-accent-dark)`）
  - **学员实操**：重构自己的 style.css（15分钟）

**内容建模与动态渲染（进阶技巧）**

- **为什么要内容建模**
  - 对比演示：
    - 硬编码：在HTML中写死10个作品
    - 数据驱动：JSON + JavaScript动态生成
  - 好处：易于更新、易于扩展、代码量更少

- **创建数据文件**
  - 教师演示创建 `assets/data.js`：
    ```javascript
    const artworks = [
      {
        id: "work-001",
        title: "春天的猫",
        image: "./assets/images/work-001.jpg",
        description: "1930年代橡皮管风格的猫咪插画",
        year: 2024,
        tags: ["插画", "复古"]
      },
      // ... 更多作品
    ];
    ```

- **JavaScript 动态渲染**
  - 读取数据
  - 动态创建HTML元素
  - 插入到页面中
  - **学员实操**（可选）：尝试将一个页面改为动态渲染

**响应式设计与无障碍性优化**

- **响应式基础（如果时间允许）**
  - CSS Media Queries 简介
  - 移动端适配的最小实践
  - 演示：让导航栏在小屏幕上可用

- **无障碍性验收**
  - 使用浏览器 Lighthouse 工具
  - 检查项目：
    - 所有图片有 alt 属性
    - 颜色对比度符合 WCAG 标准
    - 键盘可以导航所有交互元素
    - Focus 状态清晰可见
  - **学员实操**：运行 Lighthouse 并修正问题

**部署到 GitHub Pages**

- **部署流程演示**
  1. 在 GitHub 创建仓库（命名为 `username.github.io` 或项目名）
  2. 将本地代码推送到 GitHub
  3. 在仓库设置中启用 GitHub Pages
  4. 访问部署的网站：`https://username.github.io/项目名`

- **学员实操**（如果有时间）
  - 跟随教师步骤部署自己的网站
  - 可以课后完成

**系统进化思维与课程总结**

- **版本化与演进计划**
  - 当前版本：v1.0（基础功能完成）
  - 在 README.md 中规划未来：
    ```markdown
    ## 版本规划
    
    ### v1.0（当前版本）
    - ✅ 四个核心页面
    - ✅ 设计系统建立
    - ✅ 基础交互功能
    
    ### v1.1（计划）
    - [ ] 移动端响应式优化
    - [ ] 增加博客页面
    - [ ] 性能优化（图片懒加载）
    
    ### v2.0（愿景）
    - [ ] 引入 CMS 内容管理
    - [ ] 多语言支持
    - [ ] 暗黑模式
    ```

- **反思性讨论**（元认知培养）
  - 提问1：这套 AI + Figma 工作流的最大优势是什么？
    - 学员分享（2-3位）
  - 提问2：它可能不适用于哪些场景？
    - 讨论：大型企业项目、实时协作、复杂交互
  - 提问3：如果未来 AI 更强大，这个流程的哪一步最可能被取代？
    - 引导学生思考：人类的价值在于创意和决策，而非执行

- **课程总结：从"学习者"到"创造者"**
  - 你们不仅学会了技术，更重要的是：
    - 学会了如何与 AI 协作
    - 学会了系统化思维（设计系统、版本化）
    - 学会了长期主义（持续迭代而非一次性交付）
  - 鼓励：这只是开始，你的网站应该随着你的成长而进化

- **结业与展望**
  - 可选进阶路径：
    - 学习 JavaScript 框架（React / Vue）
    - 深入无障碍性和性能优化
    - 探索 3D Web（Three.js / A-Frame）
  - 社区资源分享：GitHub 学习路径、设计资源网站

## 🎯 关键成功因素

### 教学层面

1. **严格控制课堂时长**
   - 每节课不超过 90 分钟
   - 保护学员注意力和心流状态
   - 预留缓冲时间应对突发问题

2. **强制性课间作业**
   - 明确交付物和验收标准
   - 第一周作业是整个课程的基石
   - 设置作业检查点和互助机制

3. **同伴反馈机制**
   - 每节课都有学员展示环节（2-3位）
   - 培养批判性思维和沟通能力
   - 通过观摩他人作品获得灵感

4. **教师示范质量**
   - 提前测试所有流程，避免现场卡顿
   - 准备 B 计划应对工具故障（如 GitHub 访问、AI 超时）
   - 真实展示问题解决过程，而非完美演示

### 学员层面

1. **多样性管理**
   - 设计分层作业：必做 + 可选
   - 为零基础学员提供额外辅导时间
   - 为进阶学员提供挑战任务

2. **技术支持**
   - 建立课程 GitHub Discussions 或微信群
   - 提供常见问题 FAQ 文档
   - 鼓励学员互助解决技术问题

3. **激励机制**
   - 展示优秀学员作品
   - 强调"完成比完美更重要"
   - 庆祝每个里程碑（网站上线、设计系统完成等）

## 💡 差异化价值主张

相比传统 Web 开发课程，本课程的独特价值：

### 1. AI First（AI 优先）
- **传统课程**：先学语法和概念，后期才接触 AI 辅助工具
- **本课程**：从第一天就与 AI 协作，AI 是团队成员而非工具

### 2. 逆向工程方法论
- **传统课程**：从理论到实践（设计系统 → 设计稿 → 代码）
- **本课程**：从成果到规范（快速生成网站 → 反推设计系统）
- **优势**：更符合认知规律，激发兴趣后再深入原理

### 3. 系统思维培养
- **传统课程**：教代码片段和单点技术
- **本课程**：教设计系统、工作流、版本化思维
- **目标**：从"代码工匠"成长为"系统架构者"

### 4. 长期主义导向
- **传统课程**：完成作业即结束
- **本课程**：强调可维护性和持续进化
- **理念**：网站是有生命的产品，应随着创作者成长而迭代

### 5. 角色扮演与认知升级
- **传统课程**：学生 = 执行者
- **本课程**：学生 = 产品负责人 + AI 团队的编排者
- **价值**：培养产品思维和团队协作能力
