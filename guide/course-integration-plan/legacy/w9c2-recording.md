## 📚 重要术语和人物

- **Figma**：一款基于浏览器的协作式UI/UX设计工具，设计师可以用它来创建网站、移动应用等数字产品的原型和界面设计。
    
- **GitHub**：一个面向开源及私有软件项目的托管平台，也是一个代码版本控制和协作的平台，开发者在此存储、管理、分享和协作开发代码。
    
- **Design Tokens (设计令牌)**：在设计系统中，用于存储和管理设计决策（如颜色、间距、字体大小等）的变量。它们使得设计和开发能够保持一致性，并能快速进行全局样式更新。
    
- **PRD (产品需求文档)** (Product Requirements Document)：一份详细描述产品功能、特性、目标用户和业务需求的文档，是产品开发团队（设计、开发、测试）共同遵循的指南。
    
- **AI Agent (人工智能智能体/代理)**：在这个语境下，指的是为特定任务定制的高度专业化的AI提示（Prompt），例如“分析师智能体”专门负责需求分析，“开发者智能体”专门负责编写代码。
    
- **WebP**：由Google开发的一种现代图像格式，旨在为Web上的图像提供更优的无损和有损压缩效果。与PNG和JPEG相比，WebP格式的图片通常文件更小，但质量相当。
    
- **React / Vue**：两个非常流行的前端JavaScript框架，用于构建复杂的用户界面和单页应用程序。它们提供了组件化的开发模式，提高了开发效率和代码的可维护性。
    
- **WCAG (Web内容无障碍指南)** (Web Content Accessibility Guidelines)：一系列旨在让残障人士（如视力、听力、运动障碍者）能够更方便地访问和使用Web内容的指导原则和标准。
    

## 💡 我的学习收获

### 我学到的核心内容

我从这个视频中学到了一个现代化的、从零到一构建个人网站的完整工作流程。这个流程不再是传统地直接埋头设计或编码，而是巧妙地整合了人工智能（AI）来加速前期规划，并利用专业设计工具（Figma）来构建一个可维护、可扩展的设计系统。整个流程可以概括为：**AI辅助规划 -> Figma系统化设计 -> 编码实现**。

### 我的理解和思考

我认为最重要的观点是 **“流程先于工具，系统大于碎片”**。

1. **AI作为规划加速器**：视频中最具启发性的一点是将AI定义为不同角色的“智能体”（Agent），如分析师、产品经理、设计师等。通过给AI设定明确的角色和任务（例如生成项目简报、产品需求文档），可以将模糊的想法快速结构化、文档化。这不仅极大地提高了前期规划的效率，也保证了项目从一开始就有清晰的方向。
    
2. **Figma作为设计和开发之间的桥梁**：Figma在这里不仅仅是一个画图工具，它通过“设计令牌”（Design Tokens）建立了一个严谨的设计系统。将颜色、字体、间距等定义为变量，使得设计稿本身就具有了代码的逻辑性和可维护性。当需要修改品牌色时，只需改动一个变量，整个设计便能一键更新，这种系统化的思维对于后续的开发和长期维护至关重要。
    
3. **编码是实现的最后一步**：在所有规划和设计都完成后，编码实现就变得水到渠成。由于有了详细的文档和规范的设计系统，开发过程中的不确定性大大降低，可以更专注于功能的实现。
    

### 我的实践计划

我计划将这个工作流应用到我自己的个人作品集网站项目中：

1. **定义“智能体”**：我会先创建几个Markdown文件，分别定义“项目分析师”、“UI/UX设计师”和“前端开发者”的角色、知识背景和任务指令。
    
2. **AI生成初始文档**：使用定义好的“分析师”和“产品经理”智能体，与AI（如DeepSeek或Gemini）对话，生成项目的初步简报（Brief）和产品需求文档（PRD），明确网站的目标、风格和功能。
    
3. **在Figma中建立设计系统**：根据AI生成的文档，在Figma中着手设计。我会优先建立一个包含颜色、字体和间距的设计令牌系统，然后再用这些令牌来构建具体的页面组件和布局。
    
4. **编码实现**：完成Figma设计后，我会参考设计稿和之前生成的架构文档，开始编写HTML、CSS和JavaScript代码。
    

### 我想进一步探索的

- **高级AI提示工程**：如何编写更高效、更专业的“智能体”指令，让AI能够生成质量更高的文档和设计建议。
    
- **自动化工作流**：探索是否有工具可以将Figma中的设计令牌直接导出为CSS变量，从而实现从设计到开发的无缝衔接，进一步提高效率。
    
- **组件化开发**：深入学习React或Vue这样的前端框架，看看如何将Figma中的组件化设计思想更好地应用到实际的代码开发中。

## 课程笔记
### 课程目标：从零到一构建个人网站

本次课程的核心目标是带领学员从零开始，实战构建一个功能完整的个人网站。  
课程将以一个模拟的插画艺术家个人网站为例进行全程演示。  
该示例网站包含四个核心页面：**首页 (index.html)、插画作品页 (illustrator)、衍生产品页 (product) 和关于我页面 (about)**。  
最终目标是让学员掌握独立创建和部署个人作品网站的能力。

### 核心工作流：AI规划、Figma设计与编码实现

本次实战采用一套现代化的、分阶段的网站开发流程。  
这个流程强调结构化和可维护性，确保项目从规划到实现都清晰有序。  
**第一阶段是AI辅助规划**：利用AI智能体（Agents）进行头脑风暴，并生成项目简报（Brief）和产品需求文档（PRD）。  
**第二阶段是系统化设计**：在Figma中创建详细的UI/UX设计稿，并建立一个以设计令牌（Design Tokens）为核心的设计系统。  
**第三阶段是编码实现**：基于前期的文档和设计系统，使用基础的前端技术（HTML, CSS, JS）编写代码并完成网站功能。

### 阶段一：AI智能体（Agent）辅助项目规划

在项目初期，我们不直接开始设计或编码，而是首先利用AI进行深度规划。  
为此，我们为AI定义了多个专业角色，即“**智能体 (Agents)**”，每个角色都有明确的指令文件（Prompts）。  
例如，我们会分别与“**分析师 (Analyst)**”和“**产品经理 (Product Manager)**”等智能体对话。  
通过与这些智能体协作，我们可以快速生成项目所需的关键文档，如**项目简报、需求文档（PRD）、设计规范和技术架构**。  
所有这些由AI生成的规划文档，都将被统一存放在项目的 docs 文件夹中，作为整个开发过程的指导依据。

### 阶段二：使用Figma构建可维护的设计系统

在完成AI规划后，我们进入视觉设计阶段，核心工具是Figma。  
Figma在此不仅是绘图工具，更是构建一个严谨、可维护的设计系统的平台。  
**Design Tokens（设计令牌）是这个系统的核心**，我们将所有的设计决策，如颜色、字体、间距等，都定义为变量。  
例如，我们会定义品牌主色、点缀色（accent color）、以及从深到浅不同层级的灰度色变量。  
使用设计令牌的最大优势在于**确保全局一致性并简化后期维护**。  
当需要调整品牌风格时，只需修改变量值，整个设计稿中所有引用该变量的元素都会自动更新，极大地提高了迭代效率。

### 阶段三：前端技术栈与编码实现

编码是实现设计的最后一步，我们将在 VS Code 编辑器中进行。  
为了降低初学者的学习曲线，本项目选择了纯粹的基础前端技术栈。  
**技术选型为 HTML, CSS, 和 JavaScript**，刻意避开了 React 或 Vue 等复杂的前端框架。  
项目的文件结构同样保持简洁明了。  
index.html 是网站的入口主页。  
assets 文件夹用于存放所有静态资源，如图片。  
样式文件（.css）和脚本文件（.js）也遵循模块化的方式进行组织。

### 网站关键特性与UI/UX设计细节

示例网站的设计包含了一些现代网页的通用特性和最佳实践。  
**导航与交互设计**：网站顶部设计了清晰的导航栏，方便用户在不同页面间跳转，同时首页的关键区域设置了明确的行动召唤（Call to Action）按钮。  
**品牌视觉识别**：整个网站围绕一个核心的“点缀色”（如案例中的红色）来构建视觉焦点，确保了品牌形象的统一和突出。  
**作品展示效果**：插画和产品页面采用了画廊（Gallery）布局，并为每个作品增加了点击后放大的“灯箱”（Lightbox）效果，提升了用户浏览体验。  
**无障碍设计（Accessibility）**：设计遵循 WCAG 标准，通过合理的颜色对比度和层级跨度，关照到色盲等有视觉障碍的用户群体。

### 课程资源与最终部署

本课程的所有教学材料、代码和AI指令都托管在指定的 GitHub 仓库中。  
学员可以通过访问该仓库链接来下载完整的项目文件（.zip 压缩包）。  
**GitHub不仅是资源中心，也是我们最终的网站部署平台**。  
通过 GitHub Pages 功能，我们可以免费发布个人网站，并且无需在国内进行复杂的备案（ICP filing）流程。  
讲师会持续更新仓库中的内容，直到课程项目完全结束。

#### **第一部分：所有专家意见的归类与分析**

- **核心观点**: 专家们（尤其是 Expert 2, 3, 4, 5）普遍认为，当前工作流中的技术实践深度不足，尤其是在设计系统、无障碍性、内容管理和部署策略上。
    
- **具体建议**:
    
    1. **深化设计系统 (Atomic Design)**: 不应仅停留在颜色变量，应扩展到完整的原子设计体系：语义令牌、尺寸、间距、排版、动效，并构建可复用的组件库（按钮、卡片等）。(Expert 2, 5)
        
    2. **强化无障碍性 (WCAG)**: 不能只停留在口号，必须引入具体、可操作的验收标准和自动化测试工具（如 axe-core, Lighthouse），特别是针对导航和灯箱等高风险组件。(Expert 2)
        
    3. **引入内容建模**: 应将作品/产品从“页面”思维转变为“数据模型”思维，使用JSON/YAML等格式定义内容结构，实现数据与视图的分离，为未来的可维护性打下坚实基础。(Expert 2)
        
    4. **优化部署策略**: 针对GitHub Pages在中国访问不稳定的问题，应引入CDN和镜像部署策略（如 Cloudflare Pages）作为补充方案。(Expert 2)
        
    5. **实现自动化**: 探索从Figma设计令牌到CSS变量的自动化导出工作流，甚至实现从AI规划到最终代码的端到端自动化管道。(Expert 3, 4)
        
- **批判性分析与筛选**:
    
    - **采纳**: 深化设计系统、强化无障碍性、引入内容建模和优化部署策略，这些是**极其重要且实践性极强**的建议。它们直接关系到学生最终产物的专业度和可维护性，完全符合课程目标。

- **核心观点**: 专家们（尤其是 Expert 1, 2, 4, 6）一致认为，当前对AI的应用模式过于工具化和线性，缺乏治理和深度协作。
    
- **具体建议**:

    - **采纳**: **培养反思性实践者**和**元认知**的建议非常重要，它关乎学生长期的成长能力，是课程价值的升华。**系统进化思维**也是一个极好的收尾视角，能赋予学生更强的项目所有感和发展观。

**1. [最重要] 深化设计系统，贯彻原子设计思想**

- **核心思想**: 将“设计令牌”从零散的颜色变量，升级为贯穿整个项目的、结构化的“单一事实来源”。
    
- **具体行动**:
    
    - **扩展令牌范围**: 在Figma中，除了颜色，系统性地建立**字体排版（字号、行高、字重）、间距（margins, paddings）、圆角、阴影**等核心设计令牌。
        
    - **建立组件库**: 指导学生基于令牌创建基础组件（原子/分子），如按钮、输入框、卡片，并明确定义其不同状态（hover, focus, active）。
        
    - **实践应用**: 在后续编码环节，严格要求学生使用从设计令牌转换来的CSS变量，杜绝“魔法数值”，真正实现设计与代码的同步。
        

**2. [高度重要] 引入内容建模，分离内容与展示**

- **核心思想**: 培养学生“数据库思维”，将网站内容（如作品信息）结构化，为未来的扩展和维护奠定基础。
    
- **具体行动**:
    
    - **创建数据文件**: 教学中引导学生创建一个 artworks.json 或 products.json 文件。
        
    - **定义模型结构**: 在文件中，为每个作品定义清晰的字段，如 id, title, image_url, description, year, tags。
        
    - **JS渲染**: 在JavaScript部分，教授如何读取这个JSON文件，并动态地将作品数据渲染到HTML页面上。这比在HTML中硬编码内容是质的飞跃。
        

**3. [高度重要] 增加“认知脚手架”与“AI批判”环节**

- **核心思想**: 在拥抱AI效率的同时，确保学生的主体思考能力不被削弱。
    
- **具体行动**:
    
    - **前置思考**: 在调用“分析师Agent”之前，增加一个强制任务：要求学生手绘或使用简单的在线白板工具，画出网站的**核心用户流程图**和**信息架构草图**。
        
    - **迭代式提问**: 将这个草图作为与AI对话的初始输入，并教授学生如何基于AI的初步反馈，进行追问、修正和深入探讨，而不是被动接受。
        

**4. [重要] 建立AI治理规范，确保教学可复现**

- **核心思想**: 将AI的使用过程科学化、规范化，便于学生自查、复盘，也便于教师指导。
    
- **具体行动**:
    
    - **提供模板**: 在课程资料的 agents 文件夹中，为每个智能体提供一个 _template.md，其中包含**版本号、所用大模型、关键参数（如temperature）、输入、输出目标**等元数据字段。
        
    - **要求记录**: 要求学生在每次与AI进行重要交互（如生成文档）时，复制该模板并填写，作为项目日志的一部分。
        

**5. [重要] 强化无障碍性（WCAG）的实践与验收**

- **核心思想**: 将无障碍设计从一个抽象概念，落实为具体、可检查的代码实践。
    
- **具体行动**:
    
    - **提供Checklist**: 针对课程中的关键组件（如导航、图片、灯箱），提供一份简化的**无障碍性检查清单**。例如：“所有图片都有alt属性”、“导航链接在键盘Tab操作下有清晰的focus状态”。
        
    - **工具教学**: 简单介绍如何使用浏览器自带的 Lighthouse 工具，对最终页面进行无障碍性评分，让学生获得即时反馈。
        

**6. [推荐] 引入“元认知”教学模块**

- **核心思想**: 培养学生的韧性和批判性思维，让他们理解任何工作流都不是万能的。
    
- **具体行动**:
    
    - **设置挑战**: 在课程后半段，可以设计一个“彩蛋”任务：提供一个AI生成的、存在明显逻辑矛盾或技术错误的PRD文档，让学生小组讨论，找出问题并提出解决方案。
        
    - **课程反思**: 在课程结束时，布置一个思考题：“你认为我们这套AI+Figma的工作流，最大的优点是什么？它可能不适用于哪些场景？如果未来AI更强大了，这个流程的哪一步最可能被取代？”
        

**7. [推荐] 调整项目收尾，强调“系统进化”**

- **核心思想**: 帮助学生建立长期主义视角，将个人网站视为一个持续迭代的生命体。
    
- **具体行动**:
    
    - **版本概念**: 在部署到GitHub Pages时，明确告知学生这只是 v1.0 版本。
        
    - **未来规划**: 鼓励学生在 README.md 文件中，为自己的网站写一个简单的“未来功能规划”（Roadmap），例如“v1.1 计划增加博客功能”、“v1.2 计划优化移动端体验”。

### **[10:17] 讲解现代工作流**

好，现在我们有了AI生成的初步文档，比如项目简报（Brief）和需求文档（PRD）。接下来，我们进入Figma。

（我打开Figma文件，但首先展示的是一个空白页面和右侧的变量面板）

记住，**Figma不是我们画图的地方，它是我们为网站构建“设计宪法”的地方。**

这个宪法的核心，就是“原子设计理论”（Atomic Design）。

- **看这里，这些颜色、字体大小、间距单位，就是我们最小的、不可再分的“原子”**。它们就是我们的设计令牌。
    
- **当我们把这些原子组合起来，就得到了“分子”**。比如这个按钮，它是由一个“背景色原子”、一个“字体样式原子”和几个“间距原子”组合而成的。
    
- **再往上，多个分子组合，就构成了“有机体”**。比如我们的顶部导航栏，它就是由Logo图片和几个按钮分子组成的复杂有机体。
    

我们先构建这套系统，再用这套系统去“组装”我们的页面。这样做的好处是，未来无论网站如何扩展，它都将保持绝对的一致性和极高的维护性。
